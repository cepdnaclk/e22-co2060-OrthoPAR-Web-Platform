# Comprehensive Execution Trace: ML Model Integration

## 1. Initial Architectural Assumptions
*   **Environment**: Assumed simple installation of TensorFlow on current Python 3.14. *Correction*: Found Python 3.14 lacked stable TensorFlow wheels; shifted to Python 3.12.
*   **Model Storage**: Initially planned a simple file-system load. *Correction*: Requirement added for database-driven "is_active" flag and versioned tracking.
*   **Database**: Assumed PostgreSQL-only usage. *Correction*: Local testing environment used SQLite, leading to UUID type rendering conflicts.
*   **Inference**: Assumed high computational overhead. *Prediction*: Potential for >10s latency.

## 3. Conflict Resolution

### Conflict A: SQLite UUID Rendering
*   **Symptom**: `CompileError: Compiler <SQLiteTypeCompiler> can't render element of type UUID`.
*   **Resolution**: Refactored `models.py` to use `sqlalchemy.Types.Uuid` which yields `CHAR(32)` on SQLite and `UUID` on PostgreSQL.

### Conflict B: Pydantic Protected Namespace
*   **Symptom**: `UserWarning: Field "model_version" has conflict with protected namespace "model_"`.
*   **Resolution**: Explicitly disabled protected namespaces in `schemas.py` for the `ParScoreResponse` model.

### Conflict C: TestClient Dependency Mismatch
*   **Symptom**: `TypeError: Client.__init__() got an unexpected keyword argument 'app'`.
*   **Resolution**: Identified `httpx 0.28.1` was incompatible with `Starlette 0.27.0`. Downgraded `httpx` to `0.24.1` in the `venv_ml` environment.

### Conflict D: Windows File Locking & Paths
*   **Symptom**: `PermissionError: [WinError 32]` during test cleanup and `Model file not found` due to relative paths.
*   **Resolution**: Implemented `engine.dispose()` and `time.sleep()` retries for test cleanup. Updated `ml_service.py` to calculate absolute paths via `os.path.abspath(__file__)`.

## 5. Environment Isolation and Portability

### Database Agnosticism
*   **Design Decision**: Transitioned the codebase from PostgreSQL-specific `UUID` dialects to the cross-compatible **`sqlalchemy.Uuid`** type.
*   **Outcome**: The system now supports both **PostgreSQL** (Production/Standard Development) and **SQLite** (CI/CD and High-Velocity Testing) without manual schema modifications.

### Testing Strategy
*   **Architecture**: Leveraged file-based SQLite databases for the ML integration suite to achieve zero-overhead cleanup and isolation.
*   **Verification**: This allowed for rapid iteration and performance benchmarking of the 3D landmark detection mathematics without the latency or complexity of a full PostgreSQL container/server during the development phase.

### Schema Parity
*   **Compatibility**: Verified via dialect-specific rendering tests that the current `models.py` maintains 100% parity with **native PostgreSQL** deployments. The `sqlalchemy.Uuid` correctly renders as a native `UUID` column in PostgreSQL while gracefully falling back to a `CHAR(32)` variant in SQLite.
